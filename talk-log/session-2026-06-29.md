## What we did

### DNS seed — seed.fuckyyourcoins.com (live)

Set up sipa/bitcoin-seeder on node 2 as a DNS seed for the Defcoin network.
Three bugs required fixes before it worked:

1. **Seeds with `:port` suffix** — seeder's LookupHost ignores the port in
   `-s ip:port` form. Pass plain IPs; port comes from `--p2port` only.
2. **User agent rejection** — DefcoinCoreNu rejects inbound connections whose
   user agent doesn't start with "Defcoin". Patched
   `/bitcoin-seeder:0.01/` → `/DefcoinCore-seeder:0.01/` in `bitcoin.cpp`.
3. **setcap wiped by rebuilds** — each `make` creates a new inode, losing the
   capability. Replaced with `AmbientCapabilities=CAP_NET_BIND_SERVICE` in the
   systemd unit so it's granted at process start.

DNS setup: fuckyyourcoins.com moved to Cloudflare (Hover doesn't support
subdomain NS delegation). NS record `seed.fuckyyourcoins.com NS ns1seed...`
with glue `A: ns1seed... → 20.125.148.47`. Confirmed working with 8.8.8.8 and 1.1.1.1.

`seed.fuckyyourcoins.com` added to `vSeeds` in `chainparams.cpp` on node 1 and
DefcoinCoreNu rebuilt.

### Magic bytes investigation

The dc903.org pool operator flagged that our orphaned blocks are caused by a
split network: old miners/wallets use `fbc0b6db` magic (DefcoinCore:1.0.0),
new nodes use `defc014e` (DefcoinCoreNu:26.x). Blocks found by old-magic miners
don't propagate to new-magic-only nodes and vice versa.

**August 1, 2026** is the hard enforcement cutoff in DefcoinCoreNu — after that
date, legacy magic peers are rejected entirely. This is documented in the
upstream README and the v26.6.8-alpha release notes.

Current strategy:
- Node 1 (pool): stay in compatibility mode (`acceptlegacymagic` defaults true)
  so pool blocks propagate to all miners regardless of magic.
- Node 2 (relay): `acceptlegacymagic=0` — builds a clean defc014e peer set
  for post-cutoff operation.

**Attempted** setting `acceptlegacymagic=0` on node 1 — caused OOM during
defcoind restart, VM deallocated. Reverted. Don't set this on the pool node
until the network has mostly migrated.

### UNOMP pool stratum down after VM reboot

UNOMP started before defcoind finished loading the block index (~70s), hit
RPC error -28 (loading...), and silently gave up on port 3333. Fix:
`sudo systemctl restart defcoin-pool` after defcoind was fully loaded.

### defcoind.service crash loop — fixed on both nodes

After any reboot, defcoind was stuck in `activating (auto-restart)` loop:
- Node 1: restart counter reached 90+ after one reboot
- Node 2: restart counter reached **944** over 11 days

Root cause: stale or invalid `defcoind.pid` from the previous run caused
`Type=forking` + `PIDFile=` to fail PID parsing. systemd marked the service
failed, `Restart=on-failure` fired, each retry hit the lock from the still-
running defcoind. The real defcoind continued running fine — systemd just
couldn't track it.

Fix: `ExecStartPre=-/bin/rm -f /home/defcoin/.defcoin/defcoind.pid` in both
unit files. Also raised `TimeoutStartSec=600` (was 300/120) and added
`TimeoutStopSec=300` so defcoind has time to flush on shutdown.

To resolve: gracefully stop the running defcoind (`defcoin-cli stop`), stop the
systemd crash loop (`systemctl stop defcoind`), wait for the process to exit,
then `systemctl start defcoind`. The fixed unit prevents the loop from
recurring on future reboots.

## Orphan blocks — ongoing

Root cause: split network during magic byte migration. Until August 1, 2026,
old-magic miners can reach the pool node (compatibility mode) but can't
propagate blocks to defc014e-only nodes. Checked the upstream
defcoincore/Defcoin-Core-Nu repo — no specific orphan mitigation for pools;
the split network is treated as a migration-phase concern.

The orphan rate will resolve on August 1, 2026 when old-magic nodes are cut
off entirely (or when they upgrade beforehand).
